# ADR-002：Web 优先，桌面壳后置

| 项目 | 内容 |
| --- | --- |
| 状态 | Accepted |
| 日期 | 2026-04-24 |
| 决策 | 数据处理中心与共享能力以 Web 为主入口，桌面壳作为后续可选增强 |

## 背景

系统后续不仅要稳定采集，还要承载数据处理中心、共享视图和订阅推送。需要明确主入口形态，以避免过早把桌面壳当作架构核心。

## 决策

系统以 Web 作为主入口和主要共享界面，后续如需本地托盘、系统通知或一键启动能力，再按需增加 Tauri 或 Electron 壳层。

## 理由

| 理由 | 说明 |
| --- | --- |
| 数据中心天然适合中心化访问 | 多人共享内容、查看状态、管理订阅更适合 Web |
| 订阅推送核心在后端 | 匹配和分发依赖的是后端任务与数据模型，而不是桌面壳 |
| 可降低当前复杂度 | 先把 Python Core 和 Web 界面做稳，再决定是否加壳 |
| 后续扩展更平滑 | 只要内核 API 清晰，桌面壳可以后加，不需要重写内核 |

## 替代方案

| 方案 | 问题 |
| --- | --- |
| 现在就以 Electron 为主形态 | 会分散当前对采集治理和数据中心的投入 |
| 直接全量桌面化 | 不利于多人共享和中心化访问 |

## 结果

| 结果 | 说明 |
| --- | --- |
| 正面结果 | 当前阶段能把主要资源投入到稳定性、数据处理和订阅分发 |
| 约束 | 桌面壳如后续引入，只作为客户端入口，不重构核心系统 |
