# 技术沉淀路线图

本文档聚焦技术沉淀，不展开产品推广功能。目标是把项目从“能用的内部工具”演进为“可维护、可交接、可持续扩展的技术资产”。

## 阶段一：工程边界稳定

| 优先级 | 事项 | 目标 |
| --- | --- | --- |
| P0 | 固化架构文档 | 维护者能快速理解启动、采集、调度、报告、发布链路 |
| P0 | 固化安全边界 | 明确哪些配置和数据不能提交、不能打包 |
| P0 | 完善开发文档 | 新维护者能独立初始化、运行、测试 |
| P1 | 统一版本来源 | 包内可追踪版本、构建时间、Git commit |
| P1 | 增加 `CHANGELOG.md` | 发布变更可追踪 |

## 阶段二：质量保障自动化

| 优先级 | 事项 | 目标 |
| --- | --- | --- |
| P0 | GitHub Actions 测试 | push/PR 自动运行测试 |
| P1 | 打包 smoke test | 发布脚本至少做 dry-run 或关键文件校验 |
| P1 | 敏感信息检查 | CI 中拦截常见密钥误提交 |
| P2 | 格式化与 lint | 统一代码风格，减少低价值 review 成本 |

## 阶段三：数据与配置演进

| 优先级 | 事项 | 目标 |
| --- | --- | --- |
| P0 | 数据库迁移机制 | 引入 Alembic，避免表结构升级依赖自动建表 |
| P1 | 配置模型梳理 | 明确环境变量、`data/app.env`、页面配置的优先级 |
| P1 | 配置脱敏导出 | 便于排障时共享配置状态而不泄露密钥 |
| P2 | 数据备份恢复 | 为升级前后提供标准操作 |

## 阶段四：采集能力模块化

| 优先级 | 事项 | 目标 |
| --- | --- | --- |
| P0 | 采集策略接口标准化 | 统一输入、输出、异常和测试方式 |
| P1 | 策略注册机制 | 新站点接入不修改核心执行流程 |
| P1 | 站点 fixture 标准 | 每个策略都有可复现的 HTML/JSON 样例 |
| P2 | 采集诊断信息 | 页面展示可理解的失败原因和修复建议 |

## 阶段五：发布与运维沉淀

| 优先级 | 事项 | 目标 |
| --- | --- | --- |
| P0 | 版本化发布流程 | 每个 zip 对应 Git 标签和变更记录 |
| P1 | 包内版本展示 | 页面、日志或启动器能显示当前版本 |
| P1 | 升级前检查 | 检测配置、数据库、浏览器依赖和端口占用 |
| P2 | 安装器 | 在推广阶段再考虑 NSIS/Inno Setup |

## 暂不优先事项

| 事项 | 原因 |
| --- | --- |
| 前后端分离 | 当前服务端 HTML 足够支撑本地工具，先沉淀后端边界 |
| 分布式任务队列 | 当前固定电脑运行场景不需要 Redis/Celery |
| 多用户权限 | 推广或平台化前再设计，避免过早复杂化 |
| 自动绕过站点风控 | 不作为目标，应遵守站点规则和账号安全边界 |
