# ADR 0001：保留当前核心技术栈，优先工程化沉淀

| 状态 | 日期 |
| --- | --- |
| 已接受 | 2026-04-23 |

## 背景

项目当前定位是本地优先的热点信息采集工具，核心能力包括：

| 能力 | 当前实现 |
| --- | --- |
| Web 服务 | FastAPI |
| 数据访问 | SQLAlchemy |
| 默认存储 | SQLite |
| 浏览器采集 | Playwright |
| 报告导出 | Markdown + DOCX |
| 打包交付 | PyInstaller `onedir` |
| 运维脚本 | PowerShell |

在讨论长期维护和推广前，需要判断是否更换框架或重构技术栈。

## 决策

短期不更换核心技术栈，优先补齐工程化能力、文档、测试、发布、安全边界和数据迁移能力。

## 理由

| 技术 | 保留原因 |
| --- | --- |
| FastAPI | 启动快、依赖少，适合本地工具和后续 API 化 |
| SQLAlchemy | 已覆盖 SQLite/MySQL 场景，适合继续引入迁移机制 |
| SQLite | 零部署，符合固定电脑本地运行场景 |
| Playwright | 对动态页面和登录态复用有明确价值 |
| PyInstaller | 已有可用打包流程，适合 Windows 目录包交付 |
| PowerShell | 当前运维和打包环境以 Windows 为主 |

## 不采用的方案

| 方案 | 暂不采用原因 |
| --- | --- |
| React/Vue 前后端分离 | 当前页面复杂度不高，会增加构建链路和部署成本 |
| Electron/Tauri 桌面壳 | 会显著增加技术面，当前启动器已能满足本地使用 |
| Redis/Celery 分布式任务 | 当前任务量和部署方式不需要外部队列 |
| 全面 SaaS 化 | 需要用户、权限、多租户、部署运维，超出当前技术沉淀阶段 |

## 后续触发条件

| 条件 | 可能调整 |
| --- | --- |
| 多人同时在线使用 | 考虑用户体系、权限和前后端分离 |
| 大量并发采集 | 考虑任务队列和分布式 worker |
| 跨平台安装推广 | 考虑安装器或桌面壳 |
| 数据结构频繁演进 | 引入 Alembic 作为优先动作，而不是更换 ORM |

## 影响

| 影响 | 说明 |
| --- | --- |
| 正向 | 降低重构风险，把精力放在可维护性和交付稳定性 |
| 正向 | 保持现有测试和打包流程可复用 |
| 约束 | 后续新增能力需尊重当前目录边界和安全边界 |
| 风险 | 如果场景快速转向多人 SaaS，需要重新评估架构 |
