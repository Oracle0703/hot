# 采集源国内国外分组与手动分组采集设计

| 项目 | 内容 |
|---|---|
| 日期 | 2026-04-03 |
| 主题 | 采集源手动分组为国内/国外，首页立即采集拆成两个按钮 |
| 目标 | 让运维可以手动给采集源归类，并在首页分别触发国内或国外来源的手动采集 |
| 不在范围内 | 不改定时调度；不自动猜测站点归属；不新增更多分组类型 |

## 1. 背景与问题

当前系统的采集源是一个统一列表，首页也只有一个“立即采集”按钮。这样会带来两个问题：

| 问题 | 说明 |
|---|---|
| 运行范围不透明 | 点击一次“立即采集”会把所有启用来源都跑掉，不能按业务场景只跑国内或国外 |
| 运维视图混杂 | `/sources` 页面没有分组，来源一多时难以快速看清哪些属于国内、哪些属于国外 |

用户要求是：

| 要求 | 结论 |
|---|---|
| 国内/国外必须手动归类 | 采用显式字段，不做隐式规则 |
| 首页立即采集拆成两种 | 增加“立即采集国内”“立即采集国外”两个按钮 |
| 定时调度不改 | 调度任务继续跑全部启用来源 |
| 来源管理页也要分开显示 | `/sources` 页面拆分为国内、国外、未分组三块 |
| 调度页增强导航 | `/scheduler` 页面增加“返回首页”按钮 |

## 2. 设计目标

| 目标 | 设计要求 |
|---|---|
| 分组明确 | 采集源数据层必须保存“国内/国外/未分组”的状态 |
| 行为可预期 | 首页两个按钮只跑对应分组，不夹带未分组来源 |
| 旧数据安全 | 现有来源不默认归类，避免系统擅自猜错 |
| 运维可操作 | 来源创建、编辑、列表页都要明确展示分组 |
| 向后兼容 | 旧的全量手动运行和定时调度逻辑继续可用，逐步过渡 |

## 3. 方案选择

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| A | 为 `Source` 增加手动分组字段，首页双按钮按分组执行 | 清晰、可扩展、最贴合需求 | 需要改模型、接口、页面、任务创建逻辑 | 采用 |
| B | 靠站点名写死国内/国外归类 | 改动小 | 容易误判，不符合“手动归类” | 不采用 |
| C | 只在页面临时显示分组，不改数据层 | 表面简单 | 数据和执行逻辑脱节，后续维护混乱 | 不采用 |

## 4. 数据模型设计

### 4.1 Source 新字段

建议给 `Source` 增加 `source_group` 字段。

| 字段 | 类型 | 允许值 | 说明 |
|---|---|---|---|
| `source_group` | `str \| None` | `domestic`、`overseas`、`None` | `None` 表示旧数据或尚未归类 |

页面显示映射：

| 存储值 | 页面文案 |
|---|---|
| `domestic` | 国内 |
| `overseas` | 国外 |
| `None` | 未分组 |

### 4.2 旧数据策略

| 决策 | 说明 |
|---|---|
| 不自动填默认值 | 现有来源保持 `None` |
| 明显展示未分组 | `/sources` 页面单独显示“未分组采集源” |
| 不纳入国内/国外按钮 | 首页两个分组按钮只跑已明确归类的来源 |

这样可以避免系统在没有业务确认的情况下，误把旧来源归到国内或国外。

## 5. 首页立即采集设计

### 5.1 按钮设计

首页现有单个“立即采集”按钮调整为两个并列主按钮：

| 按钮 | 行为 |
|---|---|
| `立即采集国内` | 只创建并触发国内来源的手动任务 |
| `立即采集国外` | 只创建并触发国外来源的手动任务 |

设计原则：

| 原则 | 说明 |
|---|---|
| 两个按钮同级 | 不做主次按钮，避免暗示某一类更重要 |
| 不混跑 | 一次只跑一个分组 |
| 不带未分组 | 未分组来源不应被任何一个分组按钮捎带执行 |

### 5.2 后端触发方式

推荐增加明确的按分组运行入口，而不是复用同一个 `/jobs/run` 表单参数。

| 路由 | 作用 |
|---|---|
| `/jobs/run/domestic` | 触发国内来源手动采集 |
| `/jobs/run/overseas` | 触发国外来源手动采集 |

保留现有 `/jobs/run` 作为兼容全量运行入口，供旧逻辑或内部调用继续使用。

### 5.3 空分组处理

| 场景 | 处理方式 |
|---|---|
| 国内组没有启用来源 | 不创建空任务，回首页提示“国内分组没有可采集来源” |
| 国外组没有启用来源 | 不创建空任务，回首页提示“国外分组没有可采集来源” |

## 6. 来源管理页设计

### 6.1 `/sources` 列表页

列表页拆成三个区块：

| 区块 | 内容 |
|---|---|
| 国内采集源 | `source_group == domestic` |
| 国外采集源 | `source_group == overseas` |
| 未分组采集源 | `source_group is None` |

显示原则：

| 原则 | 说明 |
|---|---|
| 未分组要显眼 | 提醒运维及时补齐归类 |
| 组内仍沿用现有来源卡片 | 降低视觉改动和维护成本 |
| 空组可显示空状态 | 明确“当前没有该分组来源” |

### 6.2 新增来源页

`/sources/new` 增加一个必填分组选项。

| 字段 | 交互 |
|---|---|
| 分组 | 单选或下拉，值为“国内”“国外” |

新建时不允许留空，避免继续产生新的未分组数据。

### 6.3 编辑来源

编辑来源时也必须展示并允许修改分组，保证归类错误时可以纠正。

## 7. 任务与执行链路设计

### 7.1 任务创建

当前 `JobService.create_manual_job()` 是按全部启用来源计数。需要扩展为支持按分组创建手动任务。

建议：

| 方法 | 行为 |
|---|---|
| `create_manual_job()` | 保持原有全量行为 |
| `create_manual_job_for_group(group)` | 只统计该分组下 `enabled=true` 的来源数量 |

### 7.2 任务执行

当前 `JobRunner.run_once()` 会取出一个 `pending` 任务，然后执行全部启用来源。要支持“任务只跑一个分组”，则任务本身需要带上运行范围信息，例如新增 `source_group` 或 `scope` 字段。

建议：

| 字段 | 值 |
|---|---|
| 任务范围字段 | `None` / `domestic` / `overseas` |

执行规则：

| 任务范围 | 实际执行来源 |
|---|---|
| `None` | 全部启用来源 |
| `domestic` | 启用且分组为国内的来源 |
| `overseas` | 启用且分组为国外的来源 |

### 7.3 定时调度

| 项目 | 处理 |
|---|---|
| 调度配置页 | 不变 |
| 调度创建任务 | 继续创建全量任务 |
| 调度执行来源 | 继续跑所有启用来源，包括未分组 |

这保证定时任务不被本次需求打断。

## 8. 用户可见反馈设计

| 场景 | 页面反馈 |
|---|---|
| 国内任务创建成功 | 回到任务详情或首页，能看到本次范围为国内 |
| 国外任务创建成功 | 回到任务详情或首页，能看到本次范围为国外 |
| 分组无来源 | 首页提示该分组暂无可执行来源 |
| 存在未分组来源 | 首页统计和来源页都显示未分组数量/区块 |

任务详情页额外要求：

| `source_group_scope` | 页面显示 |
|---|---|
| `domestic` | `执行范围：国内` |
| `overseas` | `执行范围：国外` |
| `None` | `执行范围：全部` |

建议在首页统计卡片中增加：

| 统计项 | 用途 |
|---|---|
| 国内来源数 | 让“立即采集国内”是否有意义一眼可见 |
| 国外来源数 | 同上 |
| 未分组来源数 | 提醒运维还有来源未纳入双按钮体系 |

### 8.1 调度页导航补充

| 页面 | 设计 |
|---|---|
| `/scheduler` | 在页面头部操作区增加一个“返回首页”按钮 |

这样运维在配置 Cookie、钉钉和网络规则后，可以直接回到首页继续手动触发国内/国外采集，不需要手工改地址或退回浏览器历史。

## 9. 错误处理

| 场景 | 处理 |
|---|---|
| 新建来源未选分组 | 表单校验失败，不保存 |
| 更新来源分组非法 | API 返回 422 |
| 首页点国内/国外按钮但无启用来源 | 不创建任务，页面展示提示 |
| 旧来源未分组 | 允许存在，但不纳入分组按钮执行 |
| 任务运行中某个来源失败 | 维持现有任务容错逻辑，不因分组改动而变化 |

## 10. 测试设计

| 测试层 | 覆盖点 |
|---|---|
| 单元测试 | `SourceService` 分组查询；`JobService` 按分组统计；`JobRunner` 按任务范围筛选来源 |
| API 测试 | 创建来源必须带分组；更新来源可改分组；读取来源可返回分组 |
| 页面测试 | 首页出现两个按钮；来源页分三块；新建来源页出现分组选项；空分组提示正确 |
| 页面测试 | `/scheduler` 页面出现“返回首页”按钮 |
| 集成测试 | 点击国内按钮只执行国内来源；点击国外按钮只执行国外来源；未分组来源不会被两侧带上 |

## 11. 实现边界

| 包含 | 不包含 |
|---|---|
| Source 分组字段 | 自动根据站点名归类 |
| 首页双按钮 | 定时调度拆分 |
| 来源页分组展示 | 新增更多分组类型 |
| 按分组创建手动任务 | 改动现有采集策略实现 |

## 12. 最终结论

本次采用“手动分组 + 首页双按钮 + 来源页分组展示 + 定时调度不变”的方案。

核心原则如下：

| 原则 | 结论 |
|---|---|
| 分组来源 | 明确保存到数据层 |
| 分组执行 | 只影响首页手动采集 |
| 旧数据处理 | 保持未分组并显式提示 |
| 运维体验 | 来源列表与新建编辑流程都要看到分组 |
| 风险控制 | 不在本次需求中引入自动归类或调度重构 |
