# 热点采集后台视觉改版设计

## 背景

当前页面已经具备基础功能，但整体呈现仍接近原始 HTML 页面：

- 首页缺少视觉重心与状态总览
- 来源管理、新增来源、任务详情、调度页都偏“功能可用”而非“可持续使用”
- 页面之间没有统一的视觉语言，难以形成完整后台产品感

本次改版不调整业务路由和核心流程，重点提升：

- 美观度
- 信息层级
- 后台专业感
- 长时间使用时的阅读舒适度

## 目标

- 将系统改造成“现代数据中枢 + 专业运营后台”的视觉风格
- 采用“首页深色中枢、业务页浅色专业后台”的混合方案
- 覆盖主要页面：
  - `/`
  - `/sources`
  - `/sources/new`
  - `/jobs/{id}`
  - `/scheduler`
- 在不破坏现有功能和主要页面测试的前提下，统一页面结构与样式语言

## 方案选择

本次采用“方案 2：首页深色中枢 + 内页浅色专业后台”。

对比结论如下：

| 方案 | 特点 | 结论 |
| --- | --- | --- |
| 全站深色科技风 | 冲击力强，但表单与列表阅读压力大 | 不选 |
| 首页深色 + 内页浅色 | 兼顾记忆点、专业感与可读性 | 采用 |
| 全站浅色蓝灰风 | 稳定易读，但首页辨识度不足 | 不选 |

该方案同时满足以下偏好：

- 风格方向：现代数据中枢
- 落地方式：深浅混合
- 信息密度：中密度
- 页面范围：全部主要页面

## 视觉系统

### 整体气质

整体气质定义为：

- 首页像“信号中枢”
- 内页像“成熟运营后台”
- 强调专业、清晰、耐看，而不是炫技型酷炫页面

### 色彩系统

| 角色 | 颜色方向 | 用途 |
| --- | --- | --- |
| 主背景（首页） | 深海军蓝 / 墨蓝渐变 | 营造数据中枢氛围 |
| 主背景（内页） | 浅灰蓝 / 冷白 | 保证阅读舒适度 |
| 主强调色 | 冰蓝 | 主按钮、关键数字、聚焦态 |
| 成功色 | 青绿 | 成功任务、启用状态 |
| 警告色 | 橙色 | 警示、处理中 |
| 失败色 | 红色 | 错误、失败状态 |
| 中性色 | 灰蓝系 | 文本、边框、辅助说明 |

### 版式规则

| 项目 | 规则 |
| --- | --- |
| 信息密度 | 中密度，保留呼吸感，但避免稀疏到像展示页 |
| 卡片 | 大圆角、轻描边、弱阴影 |
| 间距 | 统一放大，形成成熟后台的秩序感 |
| 标题 | 更有力量感，拉开与正文层级 |
| 数字指标 | 明显放大，优先成为视觉焦点 |
| 背景 | 不使用大片纯白；首页可叠加轻微光晕或网格 |

### 交互原则

| 场景 | 表现 |
| --- | --- |
| 按钮 hover | 亮度变化或轻微上浮 |
| 卡片 hover | 阴影与边框轻微增强 |
| 输入框聚焦 | 冰蓝色聚焦态 |
| 状态反馈 | 用统一 badge，而不只靠纯文本 |
| 自动刷新区 | 过渡平滑，避免闪烁感 |
| 空状态 | 使用完整卡片提示，而不是裸文本 |

## 页面设计

### 首页 `/`

定位：信号中枢首页。

布局方案：

- 顶部使用深色 Hero 区
- 左侧显示系统标题与一句简短说明
- 右侧放主操作按钮，例如“立即采集”
- Hero 下方放 3 到 4 个核心指标卡
- 第二屏采用双栏布局：
  - 左侧：最近任务
  - 右侧：快捷入口、系统状态

重点目标：

- 进入首页时立刻感知系统状态
- 强化“这个系统是活的、可控的”
- 相比当前纯文本首页，显著提升第一印象

### 来源管理 `/sources`

定位：高频操作的资源管理页。

布局方案：

- 顶部为浅色页头，包含标题、说明和新增按钮
- 主体从原先简单 `<ul>` 改为卡片式或列表卡片式来源展示
- 每条来源至少表达：
  - 名称
  - 站点
  - 抓取方式
  - URL
  - 启用状态

重点目标：

- 提升扫读效率
- 避免所有信息挤成一条裸文本
- 让来源具备“资源对象”而不是“字符串记录”的观感

### 新增来源 `/sources/new`

定位：轻引导式配置页。

布局方案：

- 顶部提供简洁说明，告知用户只需填 URL、关键词和条数
- 表单主体置于面板卡片中
- 字段按纵向统一排列
- 提交按钮与辅助说明单独成区

重点目标：

- 从“测试页风格”提升为“正式表单页风格”
- 保持当前简化表单方案，不增加不必要字段
- 让首次配置过程更顺手

### 任务详情 `/jobs/{id}`

定位：监控与结果查看页。

布局方案：

- 顶部做状态页头，显示任务 ID、当前状态、返回入口
- 主体先展示进度总览卡
- 中间使用双栏：
  - 左侧：状态、数量、成功失败统计
  - 右侧：报告下载区
- 下方展示完整日志面板

重点目标：

- 将“看进度、看日志、拿报告”三件事分层表达
- 保持自动刷新逻辑不变，但视觉上更稳定
- 让日志区更像监控台中的可读面板

### 调度页 `/scheduler`

定位：系统设置页。

布局方案：

- 顶部做状态摘要卡，突出“已启用 / 已停用”
- 下方将当前设置放入表单卡片
- 启用开关、执行时间与保存按钮形成有层级的配置区

重点目标：

- 从原始配置页提升为正式后台设置页
- 让当前状态和可编辑项更容易分辨

## 公共组件

本次改版应优先抽出可复用的页面与样式片段，而不是每个页面独立堆内联样式。

建议抽象的视觉单元如下：

| 组件 | 作用 |
| --- | --- |
| Page Header | 页面标题、说明、右侧操作 |
| Stat Card | 指标卡，大数字 + 小标签 |
| Panel | 通用面板，用于表单、日志、列表、下载区 |
| Resource Card | 来源列表卡 |
| Status Badge | `success / failed / pending / running / enabled` 等状态标签 |
| Button Variants | 主按钮、次按钮、弱强调按钮 |
| Empty State | 空数据占位提示 |

## 实现约束

### 保持不变的部分

- 路由路径保持不变
- 表单提交与页面刷新逻辑保持不变
- 任务详情自动刷新逻辑保持不变
- 报告下载链接与 API 路径保持不变

### 改造边界

- 以服务端返回 HTML 的当前架构继续实现
- 不引入新的前端框架
- 优先通过公共样式与公共 HTML 片段统一页面，而不是一次性重构前后端结构

## 测试与验证策略

### 自动化验证

继续依赖现有页面集成测试，但允许以下变化：

- 页面外观和结构增强
- 新增 class、容器和视觉分组

必须保持以下测试依赖能力：

- 关键标题文案存在
- 关键链接存在
- 表单字段名称不变
- 自动刷新脚本仍工作

### 人工验证

应至少手动验证：

| 页面 | 验证项 |
| --- | --- |
| `/` | 首页是否具备中枢感与清晰入口 |
| `/sources` | 来源列表是否更易扫读 |
| `/sources/new` | 表单层级是否明显改善 |
| `/jobs/{id}` | 状态、日志、报告区是否清楚 |
| `/scheduler` | 设置区是否像正式后台配置页 |

## 影响范围

预计主要涉及：

- `app/api/routes_pages.py`
- `tests/integration/test_pages.py`

如需拆分公共样式或模板辅助函数，可能新增少量页面渲染辅助模块，但不应扩大到业务服务层。

## 非目标

本次不做以下事情：

- 不改业务流程
- 不改数据库结构
- 不改 API 协议
- 不引入 SPA 或组件框架
- 不新增复杂前端构建链路

## 结论

本次视觉改版采用“首页深色中枢 + 内页浅色专业后台”的混合路线，以现代数据中枢为视觉方向，中密度信息排布为基础，覆盖所有主要页面。实现时优先建立统一视觉骨架，再逐页套用到首页、来源页、新增页、任务详情页和调度页，确保功能不变但产品感明显提升。
