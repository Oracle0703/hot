# 首页风格统一全站设计

## 背景

当前系统页面已经存在一套共享样式入口，但实际使用上分成了三种视觉层级：

- 首页 `/` 使用深色“信号中枢”风格
- 来源管理、任务详情、调度等页面仍使用浅色主题
- 报告列表与报告详情仍是更早期的原生 HTML 页面，尚未接入共享页面壳

这会导致用户在首页、详情页、管理页之间切换时，产生“像进入了不同系统”的割裂感。当前需求不是重新设计信息结构，而是把整站统一到首页已经确定的视觉语言下。

## 目标

- 以首页现有深色视觉为唯一基准，统一全站主要页面风格
- 保留当前路由、表单字段、业务流程、自动刷新逻辑不变
- 将报告页纳入同一套页面壳和组件体系
- 通过最小结构调整，修复不同页面之间的配色、背景、按钮、卡片、标题层级不一致问题

## 方案选择

本次采用“方案 A：首页风格统一全站”。

对比结论如下：

| 方案 | 做法 | 结论 |
| --- | --- | --- |
| 全站统一首页深色风格 | 所有主要页面都切到首页同一套深色主题和组件语言 | 采用 |
| 保留深浅双主题 | 仅统一局部卡片和按钮 | 不采用，页面割裂仍明显 |
| 增加完整主题切换系统 | 抽象双主题 token 并支持切换 | 不采用，超出当前需求 |

选择该方案的原因：

- 用户已经明确要求页面“保持一致”
- 首页视觉已经是当前系统中最成熟、辨识度最高的一套
- 现有公共样式文件已经具备深色体系，改造成本低于重做浅色方案

## 统一后的视觉原则

### 整体气质

全站统一为“深色运营中枢”风格：

- 页面背景统一采用首页的深蓝渐变与轻微氛围层
- 主要内容承载统一使用深色半透明面板
- 强调色继续使用首页已有的冰蓝色系
- 状态反馈继续沿用成功、警告、失败三色，但适配深色背景下的可读性

### 页面语言

统一后的页面应具备以下共同特征：

| 项目 | 统一规则 |
| --- | --- |
| 页面外壳 | 使用同一套 `app-shell` 深色背景 |
| 主标题区 | 使用首页同体系的页头、眉标、标题、说明文字 |
| 卡片/面板 | 使用统一圆角、描边、阴影、深色面板背景 |
| 按钮 | 统一主按钮渐变蓝、次按钮深色描边方案 |
| 状态标签 | 统一 badge 体系，不再依赖裸文本表达状态 |
| 空状态 | 使用完整占位卡片，而不是纯文本块 |

## 页面改造范围

### 首页 `/`

首页作为基准页，整体结构和视觉方向保持不变，只允许做以下轻微修正：

- 若全站统一过程中发现共享样式需要补充，应保证首页效果不退化
- 新增的公共样式必须优先兼容首页现有 Hero、指标卡、快捷入口和最近任务区

### 来源管理 `/sources`

来源管理页统一接入首页风格，但不改变信息结构：

- 保留页面标题、返回首页、新增来源入口
- 来源列表继续使用卡片式展示
- 将页面背景、按钮、卡片、辅助文案全部切到深色体系

重点是让来源管理页看起来像首页体系中的一个业务子页，而不是切换到另一套浅色后台。

### 新增来源 `/sources/new`

新增来源页继续保留简化表单方案：

- 不增加字段
- 不修改字段名
- 不改变提交路径

改造只发生在视觉层：

- 表单面板改为首页同体系深色面板
- 输入框、辅助说明、按钮改成与首页一致的交互语言
- 页头与返回入口统一纳入深色页面结构

### 任务详情 `/jobs/{id}`

任务详情页是本次统一中最重要的业务页之一，需要从“浅色监控页”切换为“深色监控台”：

- 页头继续展示任务状态和返回入口
- 进度面板、报告面板、日志面板保留当前布局
- 自动刷新逻辑保持不变
- 需要额外关注深色背景下日志、错误提示、统计卡的可读性

重点不是重排版，而是让任务详情从视觉上延续首页“运行中枢”的感觉。

### 调度页 `/scheduler`

调度页继续保持配置页定位，但需要接入统一视觉语言：

- 顶部摘要卡切为深色面板
- 定时设置表单与钉钉通知表单统一为深色配置卡
- 开关、时间输入、文本输入在深色背景下保持足够对比度

### 报告列表 `/reports`

报告列表目前尚未接入共享页面壳，本次需要补齐：

- 使用统一的 `render_page` 和公共面板渲染
- 统一页头、返回入口、下载入口样式
- 空报告状态不能再是裸文本

### 报告详情 `/reports/{id}`

报告详情页同样接入统一外壳，但保留当前 Markdown 预览方式：

- 顶部放统一页头和操作按钮
- 正文预览区域放进统一面板
- `pre` 文本区域需要适配深色配色、滚动和换行可读性

## 实现设计

### 公共主题层

本次改造优先复用 `app/ui/page_theme.py` 中已有的样式体系，而不是重新发明一套新的 UI 层。

核心调整方向：

- 让通用组件默认兼容深色主题
- 把目前只在首页使用的深色表现，扩展成其它页面也可直接复用的通用样式
- 为表单、日志、报告预览等深色阅读场景补充必要样式

### 页面接入层

主要改造点集中在以下页面路由：

| 文件 | 责任 |
| --- | --- |
| `app/api/routes_pages.py` | 统一 `/sources`、`/sources/new`、`/jobs/{id}`、`/scheduler` 到首页风格 |
| `app/api/routes_reports.py` | 将报告列表和报告详情迁移到共享页面壳 |
| `app/ui/page_theme.py` | 补充深色全站统一所需的公共样式 |

## 保持不变的事项

以下内容不在本次改造范围内：

- 路由路径
- 表单字段名
- 提交与跳转逻辑
- 任务详情自动刷新逻辑
- 报告下载接口
- 数据模型、服务层、数据库结构

## 风险与处理

| 风险 | 说明 | 处理方式 |
| --- | --- | --- |
| 深色表单可读性不足 | 输入框、placeholder、说明文字可能对比不够 | 单独补深色输入控件样式 |
| 报告预览区域阅读压力大 | 大段 Markdown 在深色底上易显得拥挤 | 调整 `pre` 的字体色、背景、边距和滚动样式 |
| 状态色在深色背景下失真 | 浅色页可读的 badge 未必适合深色页 | 校正 badge 的背景透明度与文字颜色 |
| 首页回归风险 | 统一主题时可能误伤首页现有效果 | 以首页为基准回看关键类名和视觉块，避免退化 |

## 验证策略

### 页面验证

至少覆盖以下页面的人工与自动回归：

- `/`
- `/sources`
- `/sources/new`
- `/jobs/{id}`
- `/scheduler`
- `/reports`
- `/reports/{id}`

### 验证重点

| 项目 | 验证点 |
| --- | --- |
| 视觉一致性 | 页面背景、按钮、卡片、标题层级是否统一为首页体系 |
| 文本可读性 | 深色背景下正文、辅助文案、表单、日志是否清晰 |
| 交互不回归 | 表单提交、返回链接、报告下载、任务自动刷新仍可用 |
| 空状态表现 | 无来源、无报告、无任务时是否仍有完整样式 |
| 响应式 | 移动端下页面布局不崩坏 |

## 结论

本次改造不是新的视觉方向探索，而是一次明确的“主题收口”工作。执行完成后，首页、详情页、管理页、报告页将统一为首页现有的深色风格，形成单一、连续、可辨识的后台视觉体系。
