# 全局单报告增量更新设计

## 背景
当前系统在每次任务完成后，都会按任务 ID 生成一份新的 Markdown/DOCX 报告，并在报告列表中累计展示。这适合追溯单次任务，但不适合“长期固定来源、持续增量追踪”的使用方式。

本次设计改为：全系统只维护一份总报告。每次任务执行后，不再创建新的报告文件，而是基于历史条目状态重建同一份总报告内容，并覆盖同一路径下的 Markdown/DOCX 文件。

## 目标

| 项目 | 目标 |
|---|---|
| 报告数量 | 全系统始终只保留 1 份总报告 |
| 新内容提示 | 仅当条目是“本次任务首次进入系统”时显示 `NEW` |
| 历史内容保留 | 历史已抓取条目继续保留在总报告中 |
| 本次未抓到 | 历史条目若本次任务未命中，显示“本次未抓到” |
| 输出格式 | 继续输出 Markdown 和 DOCX |
| 页面访问 | 继续通过报告列表/详情页访问，但列表最终只有一份总报告 |

## 非目标

| 项目 | 说明 |
|---|---|
| 多报告归档 | 本次不再保留“一任务一报告”的历史归档能力 |
| 多租户/多分组报告 | 本次不按来源、日期或用户拆分多份总报告 |
| 可配置的 NEW 保留天数 | 本次不支持 NEW 持续多天或多轮 |
| 条目自动过期删除 | 本次不删除历史条目，只做“本次未抓到”标记 |

## 当前问题

| 当前实现 | 问题 |
|---|---|
| `ReportService.generate_for_job()` 每次按 `job.id` 创建新文件 | 报告数量不断增加，用户需要反复切换报告 |
| `reports` 表按 `job_id` 绑定报告 | 报告模型天然是一任务一报告 |
| `CollectedItem` 模型存在，但当前执行链路未真正落库抓取结果 | 无法可靠判断条目是否首次出现或是否本轮未命中 |

## 推荐方案
采用“历史条目持久化 + 单总报告重建”的方式：

1. 每次任务采集完成后，将命中的条目写入历史条目表。
2. 历史条目按 `normalized_hash` 做全局唯一识别。
3. 在条目表中维护“首次发现时间”“最近命中时间”“最近命中任务”等状态。
4. 总报告不做文本级增量修改，而是每次根据数据库当前状态完整重建。
5. 报告页始终指向同一条报告记录和同一组输出文件。

## 方案对比

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| 方案 A | 历史条目持久化，每次重建同一份总报告 | 状态清晰、稳定、易测试 | 需要补落库与模型字段 | 采用 |
| 方案 B | 直接在现有 Markdown 上增量追加/打标 | 实现看似直接 | 容易脆弱，DOCX 难同步 | 不采用 |
| 方案 C | 单独维护报告快照表 | 扩展性较强 | 当前需求过重 | 不采用 |

## 数据模型设计

### `collected_items`
为支持增量报告，现有 `collected_items` 需要从“临时概念模型”变成真正的历史条目存储。

建议新增字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `first_seen_at` | datetime | 首次进入系统时间 |
| `last_seen_at` | datetime | 最近一次被抓到的时间 |
| `last_seen_job_id` | UUID | 最近一次命中的任务 ID |
| `last_status` | string | 最近一次在报告中的状态快照，可选 |

其中：
- `normalized_hash` 继续作为唯一键。
- `source_id` 保留，用于按来源分组。
- `job_id` 如果保留，语义应明确为“首次发现任务”或调整为 `first_seen_job_id`，避免和“最近命中任务”混淆。

### `reports`
`reports` 表需要从“一任务一报告”调整为“系统单总报告”。

可选实现中，优先推荐：

| 字段 | 处理建议 |
|---|---|
| `job_id` | 改为可空，或改成 `latest_job_id` 表示最近更新该总报告的任务 |
| `markdown_path` | 固定为单一总报告路径 |
| `docx_path` | 固定为单一总报告路径 |
| 记录数量 | 全表仅维护一条主记录 |

如果暂时不想做数据库迁移太大，也可以保留 `job_id` 字段，但将其更新为“最近一次更新总报告的任务 ID”，并只保留一条报告记录。

## 报告生成逻辑

### 生成入口
任务结束后仍由 `JobRunner` 调用 `ReportService`，但行为改为：
- 先同步/更新历史条目状态
- 再重建全局总报告

### 状态判定规则
对于某次任务 `current_job`：

| 状态 | 判定条件 | 展示标识 |
|---|---|---|
| 本次新增 | 条目在本次任务中命中，且首次发现任务就是当前任务 | `NEW` |
| 本次命中旧条目 | 条目在本次任务中命中，但首次发现任务不是当前任务 | 不加标识 |
| 本次未抓到 | 条目历史存在，但本次任务未命中 | `本次未抓到` |

注意：`NEW` 不是永久字段，而是本轮报告生成时根据“首次发现任务是否等于当前任务”动态计算。

### 报告内容结构

| 模块 | 设计 |
|---|---|
| 标题 | `# 全局热点总报告` |
| 概览 | 最近更新任务、更新时间、总来源数、命中来源数、失败来源数 |
| 主体 | 按采集源分组列出条目 |
| 条目排序 | `NEW` 优先，其次按 `last_seen_at` 倒序 |
| 状态标记 | `[NEW]`、`[本次未抓到]`、无标记 |
| 异常摘要 | 保留当前任务日志摘要，帮助识别采集异常 |

建议 Markdown 示例：

```md
# 全局热点总报告

## 报告概览
- 最近更新任务: 123
- 更新时间: 2026-03-29 10:23:00

## 来源: B站游戏区
- [NEW] 某新帖子 - 链接 - 时间
- [本次未抓到] 某历史帖子 - 链接 - 时间
- 某持续命中帖子 - 链接 - 时间
```

## 页面与交互设计

| 页面 | 调整 |
|---|---|
| `/reports` | 列表页最终只显示一条“全局热点总报告” |
| `/reports/{id}` | 详情页继续预览 Markdown 内容 |
| 首页/任务详情 | “查看报告”继续可用，但统一跳到这一份总报告 |
| 下载接口 | 保持原有 Markdown/DOCX 下载能力 |

由于任务详情目前通过 `job_id -> report_id` 查找报告，需要改为“获取全局报告 ID”，而不是按任务查一条专属报告。

## 服务拆分建议

| 文件 | 责任 |
|---|---|
| `app/services/report_service.py` | 维护总报告记录、重建 Markdown/DOCX |
| `app/workers/runner.py` | 在任务完成后触发条目同步与总报告重建 |
| `app/services/job_service.py` | 返回全局报告 ID，而不是按任务找专属报告 |
| `app/models/item.py` | 补充历史条目状态字段 |
| `app/models/report.py` | 调整为单总报告语义 |

如实现体量变大，可以考虑新增一个专门的 `app/services/report_index_service.py` 或类似服务，负责“条目状态同步”，避免 `ReportService` 过于臃肿。

## 错误处理

| 场景 | 行为 |
|---|---|
| 某个来源抓取失败 | 保留旧条目，并在本轮报告中将该来源下未命中的历史条目标为“本次未抓到”；异常摘要记录失败日志 |
| 全部来源失败 | 仍更新总报告概览和异常摘要，但不产生新的 `NEW` |
| 单个条目重复命中 | 通过 `normalized_hash` 更新历史记录，不重复新增 |
| 报告文件写入失败 | 任务应记录错误日志，避免静默失败 |

## 测试策略

### 单元测试

| 测试点 | 说明 |
|---|---|
| 首次任务生成总报告 | 新条目显示 `NEW` |
| 第二次任务命中旧条目 | 旧条目不再显示 `NEW` |
| 第二次任务未命中历史条目 | 历史条目显示“本次未抓到” |
| 总报告复用 | 多次任务后 `reports` 表仍只有一条记录 |
| 文件复写 | 总报告路径固定，文件内容随最新任务重建 |

### 集成测试

| 测试点 | 说明 |
|---|---|
| 连续执行两次任务 | 验证总报告内容从“NEW”变成普通/未抓到状态 |
| 页面入口 | 首页、任务详情、报告页都指向同一份总报告 |
| 下载接口 | Markdown/DOCX 仍能正常下载 |

## 迁移与兼容性

| 项目 | 处理建议 |
|---|---|
| 旧报告文件 | 可暂时保留磁盘文件，但页面不再列出 |
| 旧 `reports` 记录 | 可通过一次性迁移保留最新一条，或初始化时自动收敛成单条总报告 |
| 旧任务详情 | 历史任务页中的“查看报告”改为统一跳总报告，不再对应历史快照 |

## 风险与权衡

| 风险 | 说明 | 应对 |
|---|---|---|
| 丢失单次任务快照能力 | 无法再查看某个任务当时的专属报告 | 接受为本次产品取舍 |
| 状态判断出错 | 若历史条目未正确落库，会导致 `NEW` 或“本次未抓到”判断错误 | 先补测试，再落库实现 |
| 模型迁移影响较大 | 需要调整 `reports` 和 `collected_items` 语义 | 控制为最小必要字段变更 |

## 最终结论
本次采用“全局单报告 + 历史条目持久化 + 每轮完整重建”的方案。这样可以在保留 Markdown/DOCX 下载能力的前提下，实现：
- 只有一份长期可查看的总报告
- 本轮新增内容显示 `NEW`
- 历史条目在本轮未抓到时显示“本次未抓到”
- 页面入口统一、用户操作更简单
