# 最近一周热点表格页设计

| 项目 | 内容 |
|---|---|
| 日期 | 2026-04-23 |
| 主题 | 固定网页展示最近一周爬取数据 |
| 目标 | 在现有系统内新增一个固定页面，集中展示最近 7 天采集到的热点内容 |
| 不在范围内 | 第一版不做独立前端工程、不做导出 Excel、不做复杂筛选分页、不改现有报告生成机制 |

## 1. 背景

当前系统已经具备：

| 能力 | 现状 |
|---|---|
| 采集结果入库 | `collected_items` 已保存标题、链接、发布时间、点赞、评论、播放、封面图 |
| Web 页面 | 已有首页、报告页、调度页等服务端渲染页面 |
| 钉钉通知 | 已能发送明细消息 |

用户希望补一条更稳定的查看链路：

| 诉求 | 结论 |
|---|---|
| 固定网页承载展示 | 需要 |
| 只看最近一周 | 需要 |
| 表格展示 | 需要 |
| 第一版挂在当前系统里 | 需要 |
| 实际效果可作为固定展示页使用 | 需要 |

## 2. 页面范围

第一版页面新增一个固定路由，例如 `/weekly`。

| 项目 | 设计 |
|---|---|
| 页面标题 | 最近一周热点 |
| 数据窗口 | `first_seen_at >= 当前时间 - 7 天` |
| 排序 | 优先按发布时间倒序；无发布时间时降级按首次采集时间倒序 |
| 数据来源 | `CollectedItem` |
| 入口位置 | 页面路由直接可访问；首页增加快捷入口 |

## 3. 表格结构

用户已确认展示字段如下：

| 列顺序 | 字段 |
|---|---|
| 1 | 序号 |
| 2 | 标题链接合一，可点击跳转 |
| 3 | 封面预览图 |
| 4 | 点赞数 |
| 5 | 播放量 |
| 6 | 评论数 |
| 7 | 发布时间 |

补充约束：

| 项目 | 规则 |
|---|---|
| 标题为空 | 展示“未命名内容” |
| 链接为空 | 纯文本展示标题，不包链接 |
| 封面为空 | 展示“暂无封面”占位文案 |
| 点赞/播放/评论为空 | 展示 `--` |
| 发布时间 | 优先使用 `published_at_text` 的格式化结果 |

## 4. 方案选择

| 方案 | 做法 | 结论 |
|---|---|---|
| A | 直接查数据库，服务端渲染固定表格页 | 采用 |
| B | 先做 API，再由前端异步拉取 | 暂不采用 |
| C | 每次任务生成静态 HTML | 暂不采用 |

采用 A 的原因：

| 优点 | 说明 |
|---|---|
| 改动小 | 复用现有 FastAPI 路由和页面主题 |
| 上线快 | 不需要新部署链路 |
| 数据新鲜 | 页面打开即看到最近数据库结果 |

## 5. 模块设计

建议拆成“查询服务 + 页面路由”两层：

| 模块 | 责任 |
|---|---|
| `WeeklyHotService` | 查询最近一周数据、封装排序和展示字段 |
| `routes_reports.py` | 输出 `/weekly` 页面 HTML |
| `page_theme.py` | 补表格和封面缩略图样式 |

这样可以避免把查询排序逻辑直接塞进路由函数。

## 6. 测试策略

| 类型 | 覆盖点 |
|---|---|
| 单元测试 | 最近 7 天过滤、发布时间优先排序、空字段展示降级 |
| 集成测试 | `/weekly` 页面可访问、列头正确、标题链接合一、封面预览存在、超出 7 天的数据不展示 |

## 7. 验收标准

| 项目 | 标准 |
|---|---|
| 路由 | 可通过固定地址访问最近一周页面 |
| 数据范围 | 只展示最近 7 天首次抓到的内容 |
| 列顺序 | 与用户确认一致 |
| 标题 | 文字本身可点击跳转 |
| 封面 | 直接展示缩略图 |
| 排序 | 发布时间新的排前面 |
