# 多定时调度、自定义调度分组与 B 站通知增强设计

| 项目 | 内容 |
|---|---|
| 日期 | 2026-04-23 |
| 主题 | 多个定时点、自定义调度分组、按调度分组手动运行、B 站封面图地址与钉钉通知样式增强 |
| 目标 | 让系统支持多个每天定时执行点，并按自定义调度分组执行不同来源，同时增强 B 站通知内容 |
| 不在范围内 | 第一版不支持一个来源属于多个调度分组；第一版不支持每个计划直接绑定具体来源列表；第一版不下载封面图文件到本地，仅保存封面图地址 |

## 1. 背景与问题

当前系统已经支持：

| 能力 | 现状 |
|---|---|
| 定时调度 | 每天只支持一个 `daily_time` |
| 手动分组运行 | 已支持 `domestic / overseas` 两类手动运行 |
| B 站通知 | 已支持标题、发布时间、点赞、评论、播放，但还未携带封面图地址 |

现有问题集中在两个方面：

| 问题 | 说明 |
|---|---|
| 单定时点不够用 | 运营侧希望支持多个每天执行点，例如 `08:00`、`18:00` |
| 定时范围不够灵活 | 运营侧希望不同时间点执行不同来源集合，而不是所有启用来源统一跑 |

用户确认的需求约束如下：

| 维度 | 已确认结论 |
|---|---|
| 多定时点 | 需要 |
| 绑定对象 | 绑定自定义调度分组，不先做精确来源列表 |
| 来源归属 | 一个来源先只属于一个调度分组 |
| 未分组来源 | 不参与任何定时任务，只能手动跑 |
| 分组复用 | 同一个调度分组可以挂多个时间点 |
| 手动运行 | 保留现有入口，同时新增按调度分组运行入口 |
| B 站封面图 | 需要把封面图地址一起带上并入库 |
| 钉钉标题/链接 | 标题和链接合一，显示 `［标题］（链接）` 可点击跳转 |

## 2. 设计目标

| 目标 | 设计要求 |
|---|---|
| 调度灵活 | 支持多个定时点，每个定时点绑定一个调度分组 |
| 规则清晰 | 未分组来源不参与定时任务，避免误抓 |
| 兼容现有系统 | 旧的全量手动运行和国内/国外运行继续保留 |
| 数据可复用 | B 站封面图地址、点赞、评论、播放等数据入库，供通知和报告复用 |
| 页面可操作 | 运维可在 `/scheduler` 和 `/sources` 完成调度维护 |
| 第一版风险可控 | 不在第一版引入多对多绑定和复杂计划编排 |

## 3. 方案选择

### 3.1 定时模型方案

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| A | 在 `scheduler_settings` 上增加多个 `time1/time2/time3` 字段 | 表面改动少 | 扩展性差，页面和校验很快失控 | 不采用 |
| B | 新增 `schedule_plans` 表，一条记录代表一个定时点 | 结构清晰，天然支持多个定时点 | 需要新增模型和页面 | 采用 |
| C | 每个计划直接绑定具体来源列表 | 最灵活 | 明显超出第一版范围，复杂度高 | 暂不采用 |

### 3.2 来源归属方案

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| A | 每个来源新增一个 `schedule_group` 字段 | 简单、稳定、易维护 | 一个来源暂时只能属于一个组 | 采用 |
| B | 来源与调度分组做多对多 | 最灵活 | 页面和执行逻辑复杂度明显升高 | 第二版再考虑 |

### 3.3 未分组来源处理方案

| 方案 | 做法 | 结论 |
|---|---|---|
| A | 未分组来源不参与定时任务，只能手动运行 | 采用 |
| B | 未分组默认参与“全部定时” | 容易误抓，不采用 |
| C | 强制所有来源必须分组 | 会影响现有数据兼容，不采用 |

## 4. 数据模型设计

### 4.1 保留表：`scheduler_settings`

继续保留该表，但职责收缩为“全局调度开关类配置”。

| 字段 | 作用 |
|---|---|
| `enabled` | 是否启用调度循环 |
| `daily_time` | 仅用于兼容旧版本迁移，不再作为核心调度来源 |

### 4.2 新增表：`schedule_plans`

一条计划记录代表一个“每天某时刻触发某个调度分组”的规则。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | 主键 | 计划 ID |
| `enabled` | `bool` | 该计划是否启用 |
| `run_time` | `str` (`HH:MM`) | 每天执行时间 |
| `schedule_group` | `str` | 绑定的调度分组名 |
| `last_triggered_on` | `date \| None` | 该计划最近一次触发日期，用于防重复 |
| `created_at` | `datetime` | 创建时间 |
| `updated_at` | `datetime` | 更新时间 |

设计原则：

| 原则 | 说明 |
|---|---|
| 同一分组允许多条计划 | 例如同一个分组可以同时挂 `08:00` 和 `18:00` |
| 同一时间允许多条计划 | 例如多个不同分组都在 `08:00` 执行 |
| 一条计划只绑定一个分组 | 第一版避免过度复杂 |

### 4.3 扩展表：`sources`

为 `Source` 新增 `schedule_group` 字段。

| 字段 | 类型 | 说明 |
|---|---|---|
| `schedule_group` | `str \| None` | 来源所属调度分组；为空表示不参与定时任务 |

规则如下：

| 场景 | 规则 |
|---|---|
| `schedule_group = null` | 不参与任何定时任务 |
| `schedule_group = xxx` | 只会被绑定 `xxx` 分组的计划命中 |
| 一个来源属于多个调度分组 | 第一版不支持 |

### 4.4 扩展表：`collection_jobs`

不建议复用现有 `source_group_scope`，因为它现在承载的是“国内/国外”语义。建议新增独立字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `schedule_group_scope` | `str \| None` | 本次任务绑定的调度分组；为空表示不是调度分组范围任务 |

这样可以避免两个概念混在一起：

| 字段 | 语义 |
|---|---|
| `source_group_scope` | 国内 / 国外手动运行范围 |
| `schedule_group_scope` | 调度分组运行范围 |

### 4.5 扩展表：`collected_items`

为 B 站增强数据新增字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `cover_image_url` | `str \| None` | 封面图地址 |
| `like_count` | `int \| None` | 点赞数 |
| `reply_count` | `int \| None` | 评论数 |
| `view_count` | `int \| None` | 播放数 |

其中 `like_count / reply_count / view_count` 已有设计并已开始接入，本次把 `cover_image_url` 也纳入同一层数据模型。

## 5. 迁移策略

### 5.1 旧定时配置迁移

| 场景 | 处理方式 |
|---|---|
| 旧版本存在 `scheduler_settings.daily_time` | 启动时可自动迁移成一条默认计划：`run_time = daily_time`，`schedule_group = default` |
| 旧版本未启用调度 | 不创建默认计划也可接受 |
| 旧版本已有多条计划 | 不重复迁移 |

### 5.2 旧来源迁移

| 场景 | 处理方式 |
|---|---|
| 旧来源无 `schedule_group` | 保持 `null` |
| 旧来源已有 `source_group`（国内/国外） | 不自动映射到调度分组 |

这样可以避免把“业务地区分组”和“调度分组”强行绑死。

### 5.3 SQLite 兼容

| 项目 | 处理 |
|---|---|
| 新增字段 | 沿用现有 `ensure_schema_compatibility()` 机制补列 |
| 新增表 | 启动时 `Base.metadata.create_all()` 创建 |

## 6. 页面交互设计

### 6.1 `/scheduler`

从“单个 daily_time 表单”调整为“调度计划管理页”。

| 区块 | 内容 |
|---|---|
| 调度总开关 | 是否启用后台调度 |
| 调度计划列表 | 展示已有计划：启用状态、执行时间、调度分组、最近触发日期、操作按钮 |
| 新增计划表单 | 新增一条计划：`run_time + schedule_group + enabled` |
| 调度说明 | 明确提示“未分组来源不会参与任何定时任务” |

交互规则：

| 场景 | 规则 |
|---|---|
| 新增计划 | 允许同一分组出现多次 |
| 编辑计划 | 可修改时间、分组、启用状态 |
| 删除计划 | 只删除该计划，不影响来源配置 |
| 无计划时 | 页面显示空状态提示 |

### 6.2 `/sources`

来源新建和编辑表单新增 `schedule_group`。

| 字段 | 交互 |
|---|---|
| 调度分组 | 文本输入或下拉 + 可手填 |

展示规则：

| 场景 | 行为 |
|---|---|
| 未填写调度分组 | 页面提示“该来源不参与定时任务” |
| 已填写调度分组 | 页面显示所属分组 |

### 6.3 首页 `/`

在保留现有入口的同时，新增按调度分组运行入口。

| 入口 | 行为 |
|---|---|
| 立即采集全部 | 维持现有逻辑 |
| 立即采集国内 / 国外 | 维持现有逻辑 |
| 按调度分组运行 | 新增，根据选中的 `schedule_group` 创建任务 |

第一版推荐做法：

| 方案 | 说明 |
|---|---|
| 首页增加一个调度分组运行卡片 | 下拉选择已有调度分组，然后触发运行 |

### 6.4 任务详情页

任务详情页需要显示新的范围信息。

| 字段 | 页面显示 |
|---|---|
| `schedule_group_scope = morning-news` | `执行范围：调度分组 morning-news` |
| `schedule_group_scope = null` | 保持现有逻辑 |

## 7. 调度运行规则

### 7.1 调度扫描

`SchedulerLoop` 每轮都扫描全部启用的计划。

| 条件 | 是否创建任务 |
|---|---|
| 计划禁用 | 否 |
| 当前时间 `< run_time` | 否 |
| `last_triggered_on == 今天` | 否 |
| 对应分组没有启用来源 | 否，记日志 |
| 条件全部满足 | 是 |

### 7.2 同组复用

| 场景 | 规则 |
|---|---|
| 同一分组多个时间点 | 允许 |
| 多个分组同一时间点 | 允许 |
| 同一分钟多个计划 | 每条计划都可独立触发 |

### 7.3 防重复

按“计划维度”防重，而不是按“分组维度”防重。

| 维度 | 说明 |
|---|---|
| 计划维度 | 每条计划每天只触发一次 |
| 分组维度 | 不做全局去重，否则会错误吞掉同分组的第二个合法时间点 |

### 7.4 无来源分组

建议行为：

| 场景 | 行为 |
|---|---|
| 计划绑定的分组当前没有启用来源 | 不创建空任务，写一条日志 |

## 8. 手动运行规则

### 8.1 保持原有入口

| 入口 | 状态 |
|---|---|
| 全量运行 | 保留 |
| 国内/国外运行 | 保留 |

### 8.2 新增按调度分组运行

| 行为 | 规则 |
|---|---|
| 手动选择某个 `schedule_group` | 只运行该分组下的启用来源 |
| 分组下无启用来源 | 不创建任务，页面提示 |

### 8.3 与现有国内/国外分组关系

| 维度 | 语义 |
|---|---|
| `source_group` | 国内 / 国外 |
| `schedule_group` | 调度分组，例如 `早报组`、`晚报组` |

两套分组并行存在，不互相覆盖。

## 9. B 站数据增强设计

### 9.1 采集结果字段

`bilibili_profile_videos_recent` 采集结果增加：

| 字段 | 说明 |
|---|---|
| `cover_image_url` | 封面地址 |
| `like_count` | 点赞数 |
| `reply_count` | 评论数 |
| `view_count` | 播放数 |

### 9.2 获取方式

推荐继续沿用“先发现新视频，再按 `bvid` 调详情接口补数据”的模式。

| 数据 | 来源 |
|---|---|
| 标题 / URL / 发布时间 / 作者 | 主页列表或列表接口 |
| 点赞 / 评论 / 播放 / 封面图 | 视频详情接口 |

### 9.3 入库策略

| 场景 | 行为 |
|---|---|
| 首次采集 | 写入 `cover_image_url / like_count / reply_count / view_count` |
| 重复采集 | 用最新值覆盖旧值 |
| 钉钉发送时 | 优先读已入库值，缺失时再回退补取 |

## 10. 钉钉通知样式设计

### 10.1 标题规则

| 规则 | 结果 |
|---|---|
| 来源名已包含 `热点报告` | 直接使用 |
| 来源名不含 `热点报告` | 使用 `热点报告 {来源名/UP 名}` |

### 10.2 正文样式

每条内容采用如下结构：

```text
### 热点报告 初夏ChuXXia

1. [PRAGMATA/识质存在：恰到好处的力竭](https://www.bilibili.com/video/BV1z4oWB3Ex9)

发布时间：2026-04-22 11:57 | 点赞：3689 | 评论：206 | 播放：61317

封面图：https://i2.hdslb.com/bfs/archive/xxxx.jpg
```

设计原则：

| 原则 | 说明 |
|---|---|
| 标题和链接合一 | 使用 Markdown 链接，减少冗余 |
| 统计信息单独一行 | 避免一整行过长难读 |
| 封面图地址单独一行 | 先满足可见性，后续若要直接展示图片可再扩展 |

## 11. 错误处理

| 场景 | 处理方式 |
|---|---|
| 保存计划时 `run_time` 非法 | 表单校验失败 |
| 保存来源时 `schedule_group` 为空 | 允许，但提示不参与定时 |
| 计划触发时分组无来源 | 跳过并记日志 |
| B 站详情接口取不到封面图/统计 | 保留已有值，不阻断任务 |
| 钉钉发送时缺少封面图地址 | 正文省略封面图行 |

| 测试层 | 覆盖点 |
|---|---|
| 单元测试 | `SchedulerService` 多计划扫描、防重复、无来源跳过 |
| 单元测试 | `JobService/JobRunner` 按 `schedule_group_scope` 只跑对应来源 |
| 单元测试 | 来源表单/服务层保存 `schedule_group` |
| 页面测试 | `/scheduler` 计划列表、新增、编辑、删除 |
| 页面测试 | `/sources` 展示/编辑调度分组 |
| 页面测试 | 首页按调度分组运行入口 |
| 单元测试 | B 站策略提取并保存 `cover_image_url` |
| 单元测试 | 钉钉通知正文展示 `［标题］（url）`、统计行、封面图地址行 |

## 13. 实现边界

| 包含 | 不包含 |
|---|---|
| 多条调度计划 | 多对多来源绑定 |
| 自定义调度分组 | 每个计划直接选具体来源 |
| 未分组来源不参与定时 | 自动把旧来源分配进默认分组 |
| 手动按调度分组运行 | 下载封面图文件到本地 |
| B 站封面图地址入库和通知展示 | 直接在钉钉中嵌入图片本体 |

## 14. 最终结论

本次采用“多计划表 + 来源单调度分组 + 未分组不参与定时 + 手动按调度分组运行 + B 站封面图地址入库与通知展示”的方案。

核心结论如下：

| 维度 | 结论 |
|---|---|
| 调度模型 | 使用 `schedule_plans` 支撑多个定时点 |
| 来源归属 | 使用 `sources.schedule_group`，一来源先只归一个组 |
| 定时范围 | 每条计划只跑绑定分组下的启用来源 |
| 手动运行 | 保留旧入口，新增按调度分组运行 |
| B 站增强 | 增加 `cover_image_url` 并与统计字段一起入库 |
| 钉钉样式 | 使用 `［标题］（url）` + 统计行 + 封面图地址行 |
