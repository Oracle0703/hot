# 源站抓取间隔与相对时间展示设计

## 背景

当前任务执行器会按来源串行抓取，但来源之间没有任何等待时间。随着来源数量增加，尤其是同一轮里包含多个 B 站来源时，连续请求更容易触发风控。

同时，部分来源返回的是相对时间文本，例如 `12小时前`、`3 days ago`、`5 個月前`。系统需要保留这类原始时间文本，避免误格式化。

## 目标

| 项目 | 目标 |
| --- | --- |
| 相对时间显示 | 保留原始文本，不转成伪造的具体时分 |
| 全局抓取间隔 | 支持配置所有来源之间的基础等待秒数 |
| B站专用抓取间隔 | 支持额外配置 B 站来源之间的等待秒数 |
| 配置入口 | 在 `/scheduler` 页面集中配置并写入 `data/app.env` |
| 默认行为 | 默认值为 `0`，保持现有行为不变，由用户按需开启 |

## 方案

| 方案 | 说明 | 结论 |
| --- | --- | --- |
| 只加全局间隔 | 简单，但会拖慢所有站点 | 不推荐 |
| 只加 B站间隔 | 对目标问题有效，但缺少通用控制 | 不完整 |
| 全局间隔 + B站专用间隔 | 先执行全局等待，再对 B 站追加专用等待 | 采用 |

## 配置项

| 键名 | 含义 | 默认值 |
| --- | --- | --- |
| `SOURCE_FETCH_INTERVAL_SECONDS` | 任意两个来源之间的基础等待秒数 | `0` |
| `BILIBILI_SOURCE_INTERVAL_SECONDS` | 当来源属于 B 站时追加等待秒数 | `0` |

## 规则

| 场景 | 等待规则 |
| --- | --- |
| 非首个来源 | 执行前可等待 |
| 普通来源 | 只等待 `SOURCE_FETCH_INTERVAL_SECONDS` |
| B站来源 | 等待 `SOURCE_FETCH_INTERVAL_SECONDS + BILIBILI_SOURCE_INTERVAL_SECONDS` |
| 首个来源 | 不等待，避免首轮无意义阻塞 |

## 涉及文件

| 文件 | 作用 |
| --- | --- |
| `app/config.py` | 读取新增环境变量 |
| `app/services/app_env_service.py` | 读写新增运行时配置 |
| `app/api/routes_pages.py` | 调度页展示与保存配置 |
| `app/workers/runner.py` | 在来源执行前注入等待逻辑 |
| `tests/unit/test_runner.py` | 验证等待秒数与来源类型判断 |
| `tests/unit/test_app_env_service.py` | 验证环境文件写入 |
| `tests/unit/test_config.py` | 验证配置读取 |
| `tests/integration/test_pages.py` | 验证调度页入口与提交保存 |

## 验收标准

| 编号 | 标准 |
| --- | --- |
| 1 | 相对时间文本继续原样显示 |
| 2 | 调度页可以保存全局间隔与 B站专用间隔 |
| 3 | `app.env` 会写入新增配置项 |
| 4 | 执行器会在来源之间按配置等待 |
| 5 | B站来源会额外应用专用等待 |
