# 热点信息采集系统开发计划（plan.md）

## 1. 当前基线

| 项目 | 状态 | 说明 |
| --- | --- | --- |
| Web 工作台 | 已完成 | 首页、任务详情、采集源列表、报告页、调度页可访问 |
| 采集源管理 | 已完成 | 支持 API 和表单新增、列表、更新、删除 |
| 采集执行主链路 | 已完成 | 手动创建任务、后台执行、状态回写已打通 |
| 定时调度 | 已完成 | 轻量后台轮询器按日创建 `scheduled` 任务 |
| 报告生成 | 已完成 | 自动输出 Markdown 和 DOCX |
| 数据库策略 | 已完成 | 默认 SQLite，可切 MySQL |
| 测试基线 | 已完成 | 当前 `46 passed` |

## 2. 当前架构决策

| 维度 | 方案 |
| --- | --- |
| 部署方式 | 单机部署，不使用 Docker |
| 应用形态 | 单个 FastAPI 进程，内部包含页面、API、调度和后台执行 |
| 页面实现 | 服务端 HTML + 轻量 JS 轮询 |
| 调度实现 | `SchedulerLoop + SchedulerService` |
| 执行实现 | `JobDispatcher + JobRunner` |
| 数据库存储 | SQLite / MySQL 二选一 |
| 报告输出 | 落盘到 `outputs/reports/` |

## 3. 下一阶段开发目标

| 优先级 | 主题 | 目标 |
| --- | --- | --- |
| P1 | 来源管理增强 | 增加编辑页、删除确认、字段校验、排序能力 |
| P1 | 来源校验 | 增加“试抓一次”能力，保存前即可验证选择器与抓取模式 |
| P1 | 调度可视化 | 在调度页显示 `last_triggered_on`、最近一次执行结果 |
| P1 | 失败恢复 | 增加失败来源重试、报告重新生成 |
| P2 | 登录态管理 | 为 Playwright 来源增加 storage state 受控目录和失效提示 |
| P2 | 通用性增强 | 增加规则模板、示例来源导入、特殊站点插件机制 |
| P3 | 智能能力 | AI 摘要、热点打分、消息推送 |

## 4. 分阶段任务

### 阶段 A：把运营可用性补齐

| Task | 目标 | 交付物 |
| --- | --- | --- |
| A1 | 补采集源编辑页 | `/sources/{id}` 页面、表单回填、保存逻辑 |
| A2 | 删除前确认 | 页面确认交互、误删防护 |
| A3 | 字段校验增强 | URL、抓取模式、选择器、最大条数校验 |
| A4 | 列表可读性优化 | 显示启用状态、抓取模式、关键词摘要 |

### 阶段 B：把调试与排障能力补齐

| Task | 目标 | 交付物 |
| --- | --- | --- |
| B1 | 来源试抓 | `/api/sources/{id}/validate` 或表单试抓接口 |
| B2 | 失败详情增强 | 任务页展示来源级失败原因和时间 |
| B3 | HTML/上下文留档 | 解析失败时保留必要上下文用于定位 |
| B4 | 报告重生成功能 | 对已完成任务可重新生成 Markdown / DOCX |

### 阶段 C：把 Playwright 与真实站点接入补齐

| Task | 目标 | 交付物 |
| --- | --- | --- |
| C1 | 登录态文件管理 | 统一存放目录、来源绑定、失效提示 |
| C2 | 等待策略配置 | `wait_for_selector`、超时、滚动策略 |
| C3 | 特殊站点插件 | parser/plugin 注册能力和示例模板 |
| C4 | 示例来源导入 | `scripts/seed_demo_sources.py` 或示例 JSON |

### 阶段 D：把交付文档和运维操作补齐

| Task | 目标 | 交付物 |
| --- | --- | --- |
| D1 | `.env.example` | 标准环境变量模板 |
| D2 | 启动脚本 | `scripts/bootstrap.ps1`、`scripts/run.ps1` |
| D3 | README 完整化 | 真实启动、调度、MySQL 切换、Playwright 使用说明 |
| D4 | 冒烟测试 | 从来源配置到报告生成的最小全链路测试 |

## 5. 推荐实施顺序

| 顺序 | 建议 | 原因 |
| --- | --- | --- |
| 1 | A1-A4 | 先把日常运营配置体验补齐 |
| 2 | B1-B4 | 再把排障与失败恢复补齐，减少维护成本 |
| 3 | C1-C4 | 然后接入复杂站点与真实使用场景 |
| 4 | D1-D4 | 最后补部署脚本、环境模板和冒烟保障 |

## 6. 每阶段验收标准

| 阶段 | 验收标准 |
| --- | --- |
| 阶段 A | 运营可独立完成“新增来源 -> 编辑 -> 删除/停用 -> 发起任务” |
| 阶段 B | 任一失败来源可在 3 分钟内定位原因，并支持重试或重生报告 |
| 阶段 C | 至少 2 个真实站点稳定运行，其中 1 个为 Playwright 模式 |
| 阶段 D | 新开发者可在 30 分钟内按文档完成本地启动 |

## 7. 建议测试计划

| 类型 | 内容 |
| --- | --- |
| 单元测试 | 调度逻辑、过滤规则、解析器、报告生成 |
| 集成测试 | 来源 CRUD、任务执行、报告下载、调度页面 |
| 冒烟测试 | 示例来源 -> 创建任务 -> 生成报告 |
| 真实站点验证 | 至少验证 2 个目标站点的抓取成功率 |

## 8. 当前已知风险

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 目标站点结构频繁变化 | 规则容易失效 | 引入试抓校验和 fixture 回归 |
| Playwright 登录态失效 | 复杂站点抓取失败 | 增加登录态管理与失效提示 |
| 单进程后台线程模型 | 部署方式受限 | 先单机稳定，后续如有需要再拆分 worker |
| 实际站点差异大 | 通用规则可能不够 | 保留 parser 插件扩展点 |
